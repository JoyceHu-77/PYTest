## 快速路线：最少步骤跑通

下面 5 步做完，你就已经「从 0 到 1」跑通整套（**不需要先装 Python 虚拟环境**，除非你打算本机跑 `uvicorn`）。

1. **安装并打开 [Docker Desktop](https://www.docker.com/products/docker-desktop/)**，等菜单栏图标就绪（鲸鱼不再显示 “starting”）。
2. **打开终端**（Mac：`终端.app`；Windows：`PowerShell` 或 `cmd`）。
3. **进入项目根目录**（里面有 `docker-compose.yml` 的那一层）：
   ```bash
   cd /你的路径/PYTest
   ```
4. **执行**（第一次会下载镜像，可能较慢）：
   ```bash
   docker compose up --build
   ```
5. **浏览器打开** [http://127.0.0.1:8000/docs](http://127.0.0.1:8000/docs)，在页面里找到 **`POST /items/`** → **Try it out** → 填 JSON → **Execute**。若返回 **201** 且 body 里有 `id`，说明**写入数据库成功**。

做到这里可以先庆祝一下。终端里会一直刷日志，**要停掉服务**：在运行 `docker compose` 的那个窗口按 **`Ctrl + C`**。

## 零基础概念速览

下面这一块是「后端常识包」：不要求一次背完，**遇到陌生词可以回到这里查**。与移动端对照的地方会特别标一下，方便 iOS 方向的同学类比。

---

### 客户端、服务器、你在本机扮演谁？

- **客户端（Client）**：发起请求的一方。常见有：**浏览器**、**手机 App**、**Postman / curl**。
- **服务器（Server）**：长期运行、**监听某个端口**、等待请求的程序。本教程里 **FastAPI + Uvicorn** 就是 Web 服务器；**PostgreSQL** 是数据库服务器。
- **本机开发时**：你的 Mac/PC 上**同时**可能跑着：浏览器（客户端）+ API 进程（服务器）+ 数据库（另一个服务器）。它们通过 **127.0.0.1（本机回环）** 或 **Docker 内部网络** 互相说话。

---

### HTTP、HTTPS、URL 一眼看懂

- **HTTP**：浏览器和 Web 服务之间最常用的**请求/响应**协议（明文传输）。本教程默认用 **http://** 做练习。
- **HTTPS**：在 HTTP 外加一层**加密**（TLS），线上站点几乎都是 **https://**。
- **URL** 大致可以拆成：`协议://主机:端口/路径?查询参数`

  以 `http://127.0.0.1:8000/items/?skip=0` 为例：

  | 部分 | 含义 |
  |------|------|
  | `http://` | 使用 HTTP 协议 |
  | `127.0.0.1` | **主机**：本机（见下一小节） |
  | `8000` | **端口**：找到正在提供 API 的那个程序 |
  | `/items/` | **路径**：具体是哪个「功能入口」 |
  | `?skip=0` | **查询参数**：给列表接口传「从第几条开始」 |

- **`localhost`** 与 **`127.0.0.1`**：在开发里几乎等价，都表示「本机自己」。**手机真机**访问你电脑上的服务时，不能用它们，要用电脑的 **局域网 IP**（见附录）。
- **域名、DNS、和 Compose 里的主机名 `db`** 的区别，见 [端口、域名与运行环境详解](#端口域名与运行环境详解) 第二节。

---

### HTTP 方法（动词）：GET / POST / PATCH / DELETE …

方法表示**意图**；同一路径配合不同方法，可以表示不同操作（REST 风格里很常见）。

| 方法 | 常见用途 | 是否常带请求体（Body） |
|------|----------|-------------------------|
| **GET** | **读**：查列表、查单条 | 通常不带 |
| **POST** | **新建**：创建资源 | 常带 JSON |
| **PUT** | **整体替换**某资源（本项目未用） | 常带 JSON |
| **PATCH** | **部分更新**某资源 | 常带 JSON |
| **DELETE** | **删除** | 通常不带 |

本项目的 **`/items/`** 与这些方法对应关系，见后文 **第 8 节**里的「CRUD 和 HTTP 方法对应关系」对照表。

---

### 状态码（三位数字）：一眼判断结果类型

服务器在**响应头**里带状态码；`/docs` 里 **Execute** 之后也会显示。

| 码段 | 大类 | 本教程里你会见到的例子 |
|------|------|-------------------------|
| **2xx** | 成功 | **200** OK；**201** Created（创建成功）；**204** No Content（删除成功且无返回体） |
| **4xx** | 客户端问题 | **404** 找不到资源；**422** 请求 JSON 格式/字段不符合约定 |
| **5xx** | 服务器内部错误 | **500** 常见为连库失败、代码未捕获异常等 |

**不要求背全表**，先记住：**2xx = 这次调用在协议意义上算成功**；4xx 多检查 URL、JSON、方法是否写对；5xx 看服务端日志。

---

### Header 与 Content-Type

- **Header**：请求/响应里的「元信息」，键值对形式。
- 发 **JSON 体**时，客户端一般要带：  
  **`Content-Type: application/json`**  
  否则服务器可能**读不懂** body。在 `/docs` 里试接口时，工具会自动加好。

---

### JSON 是什么？和 Swift 的关系

- **JSON**：纯文本，用来表示**对象**和**数组**。对象用 `{}`，键必须是**双引号字符串**；数组用 `[]`。
- 例：`{"title":"高等数学","description":null}`  
  - `null` 表示「空」；Python 里是 `None`，Swift 里常对应**可选类型为 nil**。
- **和 Codable 的对应**：后端 **Pydantic / OpenAPI** 描述的字段名、类型，往往就是你 iOS 里 **`Codable` 结构体**要一致的部分（命名策略可能要统一成 snake_case 或 camelCase，属进阶话题）。

---

### API、REST、资源（Resource）

- **API（应用程序接口）**：对外暴露的一组**约定**——哪些 URL、哪些方法、传什么、回什么。可以把它想成**说明书**。
- **REST**：一种**设计风格**（不是唯一标准）：用 **名词路径**（如 `/items`）+ **HTTP 方法**表达操作，**无状态**（每次请求带够信息，服务器不依赖「上一次对话」）。
- **资源**：名词所指的东西，如「一条 Item」「Item 列表」。路径 **`/items/{id}`** 里的 **`id`** 常叫**路径参数**。

---

### 关系型数据库、表、行、SQL（极简）

- **关系型数据库**（如 PostgreSQL）：数据放在**表（Table）**里，表有多**列（Column）**，每一**行（Row）**是一条记录。
- **主键（Primary Key）**：唯一标识一行，本项目里 **`items.id`** 是自增整数。
- **SQL**：结构化查询语言，例如 `SELECT * FROM items;`。本教程用 **ORM** 少写裸 SQL，但 TablePlus 里你仍可以执行 SQL 看数据。
- **持久化**：数据存在磁盘上（或 Docker **卷**里），**重启容器**后仍在；**删卷**才可能丢数据（进阶操作，需谨慎）。

---

### ORM 是什么？为什么要它？

- **ORM（Object-Relational Mapping）**：用 **Python 类**表示表、用**对象**表示行，用函数调用间接生成 SQL。
- 本项目里 **SQLAlchemy** 扮演 ORM：**`models.py`** 里 `class Item` ↔ 数据库表 **`items`**。
- **好处**：少写字符串拼接 SQL、减少部分低级错误；类型与字段集中在一处改。  
- **代价**：复杂查询有时要学 ORM 的写法；大项目表结构变更常配合 **迁移工具（如 Alembic）**。

---

### 环境变量（Environment Variable）— 提要

- **一句话**：程序读「键值对」配置，**不把密码写死在代码里**。
- 本项目里 **`DATABASE_URL`** 由 **`docker-compose.yml`** 注入 **api 容器**，**`database.py`** 用 `os.getenv` 读取；本机不设变量时用代码里的默认 `localhost` 连接串。
- **更细的**：什么叫运行环境（开发/测试/生产）、变量从哪注入、`.env` 与保密，见专节 [端口、域名与运行环境详解](#端口域名与运行环境详解)。

---

### Docker 再讲透一点

**解决什么问题**  
「在我机器能跑」往往依赖**隐式环境**（装了哪个 Python、哪个小版本、哪些系统库）。Docker 把依赖写进**镜像**，减少「换电脑就挂」的概率。

**镜像 vs 容器（再巩固）**

| 概念 | 比喻 | 说明 |
|------|------|------|
| **镜像** | 类 / 安装包 | 只读；`docker build` 或 `pull` 得到 |
| **容器** | 对象 / 跑起来的进程 | `docker compose up` 创建；可停、可删 |

**卷（Volume）**  
把数据库文件放在**命名卷**（如 **`pgdata`**）里，**删容器**默认不删卷，**数据还在**。`docker compose down -v` 会删卷，**慎用**。

**构建上下文（Build context）**  
`docker build` / compose 的 `build: .` 会把**当前目录**打包发给引擎；**`.dockerignore`** 用来排除 `.venv` 等大文件夹，加快传输。

**Compose 网络**  
同一 `docker-compose.yml` 里的服务默认在同一个**内部网络**，可以用**服务名**当主机名互连（本项目的 **`db`**）。

---

### 端口与 Docker 映射 — 提要

- Compose 里 **`"8000:8000"`** 表示 **`宿主机端口:容器内端口`**：浏览器访问本机 **8000**，Docker 转发到 **api 容器内 8000**；数据库 **5432** 同理。
- **端口冲突、多容器、URL 里何时省略端口**等，见 [端口、域名与运行环境详解](#端口域名与运行环境详解)。

---

### 后端、API、JSON（小结句）

- **后端**：提供 API、访问数据库、执行业务规则的那一侧。
- **API**：URL + 方法 +（可选）JSON 的**约定**；`/docs` 是自动生成的**可点按说明书**。
- **JSON**：客户端与 API 之间传数据的**常用格式**。

---

### 请求是怎样到达数据库的？

用一句话串起来：

**浏览器 / App** →（HTTP）→ **FastAPI（Uvicorn，端口 8000）** →（SQLAlchemy）→ **PostgreSQL（端口 5432）**

**Docker 里多记一点**：**API 容器**和 **数据库容器**是**两台逻辑上的机器**。在 API 容器里，**`localhost` 指向 API 自己**，所以要连数据库时，主机名写 Compose 服务名 **`db`**（内部 DNS 会解析到数据库容器）。你在 **Mac 浏览器**里访问的 **127.0.0.1**，指的是 **Mac**，不是容器内部。

## 端口、域名与运行环境详解

这一节把三个容易混在一起的概念拆开：**端口**（找哪个程序）、**主机名 / IP / 域名**（找哪台机器）、**运行环境与环境变量**（同一套代码在不同场合用不同配置）。读完后，你应能解释：为什么浏览器写 `127.0.0.1:8000`、而 `DATABASE_URL` 里在容器中又写 `db:5432`。

---

### 1. 端口（Port）：到底是啥？

**直观理解**  
同一台电脑、同一个 IP 地址上，可能同时跑着：浏览器、微信、你的 API、数据库……网络把数据包送到这台机器之后，还要再分发给**具体哪一个进程**。**端口**就是用来区分「这一台机器上的哪一个服务」的**整数编号**（常见范围 **0～65535**）。

**监听（Listen）与连接（Connect）**  
- **服务器进程**会**绑定（bind）**在某个 **IP + 端口**上**监听**，例如「在 `0.0.0.0:8000` 上等待 HTTP 请求」。  
- **客户端**（浏览器、App）发起连接时，要写明 **目标 IP（或主机名）+ 目标端口**，例如 `127.0.0.1:8000`。

**常见默认端口（不必全背，知道「有惯例」即可）**

| 端口 | 常见用途 |
|------|----------|
| **80** | HTTP（URL 里常**省略不写**，`http://域名` 默认就是 80） |
| **443** | HTTPS（`https://域名` 默认 443） |
| **5432** | PostgreSQL 默认 |
| **22** | SSH 远程登录 |
| **8000** | 本教程给 **FastAPI / Uvicorn** 约定的开发端口（可改，但要全局一起改） |

**同一端口不能被两个进程重复独占**  
若本机 **8000** 已被占用，再启动一个也要监听 8000 的程序会报错（如 **address already in use**）。解决思路：关掉占用的程序，或**改映射**（见下）。

**Docker 端口映射：`"8000:8000"`、`"5432:5432"`**

写法：**`"宿主机端口:容器内端口"`**（左本机，右容器）。

- 访问 **`http://127.0.0.1:8000`**：数据先到 **你 Mac/PC 的 8000**，Docker 再**转发**到 **api 容器内部的 8000**（Uvicorn 监听的是容器里的这个口）。  
- **TablePlus** 填 **`127.0.0.1:5432`**：连的是 **本机 5432**，再被转发到 **db 容器里 Postgres 监听的 5432**。

**多个容器可以同时使用「容器内 8000」吗？**  
可以。只要映射到**宿主机时左侧端口不同**，例如 `"8001:8000"` 与 `"8002:8000"` 分别对应两个 Web 容器；互不冲突的是「**宿主机上的左侧端口**」。

**改端口时要同步改哪里？**  
若把 compose 里 API 映射改成 **`"9000:8000"`**（外面 9000，里面仍是 8000）：  
- 浏览器 / App 要访问 **`http://127.0.0.1:9000`**；  
- **容器内**应用仍监听 **8000**（由 Dockerfile `CMD` 决定），一般**不必改 CMD**，只改映射左侧即可。  
若把 **Postgres 本机映射**改成 **`"5433:5432"`**，则 TablePlus 和本机 **`DATABASE_URL` 默认串**里的端口要写成 **5433**。

**排查本机端口占用（Mac 示例）**

```bash
lsof -iTCP:8000 -sTCP:LISTEN
```

---

### 2. 主机名、IP 地址与域名

**IP 地址**  
形如 **`192.168.1.10`**（IPv4）的一串数字，在网络里标识**一台主机**（严格说是某块网卡的地址）。访问服务时，最终往往要落到 **IP + 端口**。

**回环地址：`127.0.0.1` 与 `localhost`**  
- **`127.0.0.1`**（IPv4 回环）：数据**不离开本机**，在操作系统内部转给本机上监听的程序。  
- **`localhost`**：一个**主机名**，系统通常通过 **hosts 文件**把它解析成 **`127.0.0.1`**（IPv6 环境下还可能有 `::1`，入门可先忽略）。  
因此在本机浏览器访问 **`http://127.0.0.1:8000`** 与 **`http://localhost:8000`**，在开发里**几乎总是一回事**。

**域名（Domain）与 DNS**  
- **域名**：如 **`api.example.com`**，给人看的名字。  
- **DNS（域名系统）**：把域名**解析**成 **IP**，浏览器拿到 IP 再去连端口。  
线上产品通常是 **`https://你的域名`**；本机练习用 **IP + 端口** 即可，不必有域名。

**局域网 IP（内网 IP）**  
同一路由下，你的电脑可能有 **`192.168.x.x`** 这类地址。  
**手机真机**要访问你电脑上跑的 API，必须用 **电脑的局域网 IP + 端口**（如 `http://192.168.1.10:8000`），不能用 **`127.0.0.1`**（那表示**手机自己**）。

**子域名、路径、端口别混**  
- **`https://api.example.com`** 与 **`https://www.example.com`** 可以指向**不同服务器**（DNS 不同记录）。  
- 同一域名下 **`/items`** 是**路径**，**`:8000`** 是**端口**；改路径不等于改端口。

**Docker Compose 里的 `db` 不是「互联网域名」**  
在 **`DATABASE_URL: ...@db:5432`** 里，**`db`** 是 **Compose 给服务起的名字**，在 **Docker 内置 DNS** 里解析成 **数据库容器的 IP**。它**只在 compose 网络里有效**，你的手机浏览器**不能**用 `http://db:8000` 访问 API。  
**对比记忆**：

| 谁在访问 | 要连数据库时「主机」写什么 | 原因 |
|----------|----------------------------|------|
| **api 容器** | **`db`** | 与 db 容器在同一 Docker 网络，用服务名 |
| **你 Mac 上的 TablePlus** | **`127.0.0.1` 或 `localhost`** | 连的是映射到本机的端口，不是容器内 DNS |
| **你 Mac 本机跑的 Uvicorn** | **`localhost`**（若 DB 端口映射在 5432） | 进程在宿主机上，数据库端口已映射出来 |

---

### 3. 运行环境（Environment）与环境变量

**「环境」在这里指什么？**  
不是 Python 的 `venv`，而是**部署阶段 / 运行场合**：常见划分有：

| 名称 | 通俗含义 | 典型差异 |
|------|----------|----------|
| **开发（dev）** | 你自己的电脑 | 调试方便、日志详细、可用弱密码 |
| **测试（staging）** | 预上线验证 | 接近生产，数据非真实用户 |
| **生产（prod）** | 真实用户访问 | 强密码、HTTPS、监控与备份 |

**同一套代码**，在 dev 连「本机 Docker 里的库」，在 prod 连「云上的托管数据库」，**不应**为每种环境复制一份源码，而应**换配置**。

**环境变量（Environment Variable）**  
操作系统或进程启动时注入的 **`KEY=value`**。程序用 **`os.getenv("KEY")`**（Python）读取。  
**好处**：密码、数据库地址不进 Git；镜像构建一次，部署到各环境只改变量。

**常见注入方式**

| 方式 | 说明 |
|------|------|
| **Shell** | `export DATABASE_URL=...` 后启动程序（临时） |
| **docker-compose.yml `environment:`** | 本教程给 **api 容器**写 `DATABASE_URL` |
| **`.env` 文件 + Compose** | 可把变量放在 `.env`，由 compose 读取（注意勿把含密钥的 `.env` 提交仓库） |
| **云平台 / K8s** | 控制台或 Secret 注入，原理仍是环境变量或等价配置 |

**本项目中的两条线**

1. **API 在 Docker 里跑**：compose 里已写  
   `DATABASE_URL=postgresql://app:appsecret@db:5432/appdb`  
   **`db`** 是服务名；**5432** 是**容器内** Postgres 端口（db 容器内部），不是宿主机左侧映射数字。

2. **API 在本机 Uvicorn、数据库仍在 Docker**：未设置 `DATABASE_URL` 时，**`database.py`** 默认  
   `postgresql://app:appsecret@localhost:5432/appdb`  
   这里 **`localhost`** 是**你的电脑**，**5432** 是 compose **`"5432:5432"`** 映射到本机的端口。

**安全习惯（重要）**  
- **不要把生产密码写进仓库**里的 `docker-compose.yml`；公开作业可用弱口令，真上线要用**密钥管理**。  
- 教程里的 **`appsecret`** 仅适合**本地学习**。

**与「12-factor」的直觉对齐**  
十二因素应用里强调：**配置（连哪个库、用什么密钥）与代码分离**，部署时改环境，而不是改代码再打包。

### 4. 小节对照：本教程里你该用哪种组合？

| 场景 | 浏览器 / 客户端里的「主机」 | API 连数据库用的 `DATABASE_URL` 主机部分 |
|------|------------------------------|------------------------------------------|
| 本机浏览器测 API | `127.0.0.1` 或 `localhost`，端口 **8000** | （浏览器不关心这条） |
| 全在 Docker Compose | 同上访问本机映射端口 | **`db`**（在 api 容器内） |
| 本机 Uvicorn + Docker 只跑 db | 同上 | **`localhost`**（在 Mac 上） |
| 手机访问电脑上的 API | 电脑的 **局域网 IP**，端口 **8000** | 仍由 API 进程所在环境决定（同上两行之一） |

## 5. 理解项目里的 Docker 相关文件

这些文件都是**纯文本**，用 VS Code / Cursor 新建、保存即可；**Docker Desktop 不会**根据你的业务自动生成它们。

### 5.1 目录结构（和本教程相关的部分）

```
PYTest/
├── app/
│   ├── main.py              # 应用入口：建表、挂载路由、/health 等
│   ├── database.py          # 数据库地址、连接池、每次请求用的 Session
│   ├── models.py            # 表结构（ORM）
│   ├── schemas.py           # 接口上的 JSON 形状（校验用）
│   ├── crud.py              # 增删改查函数（不管 HTTP 状态码）
│   └── routers/items.py     # /items 的 URL 与 GET/POST 等
├── requirements.txt
├── Dockerfile               # 「如何制作 API 镜像」
├── docker-compose.yml       # 「起哪些容器、端口、环境变量」
├── .dockerignore            # 构建时不要拷贝哪些文件（如 .venv）
└── TUTORIAL.md              # 本教程
```

### 5.2 `Dockerfile`（只做一件事：描述 API 镜像）

- 从官方 **Python 3.12** 精简镜像开始，安装依赖，复制 `app/`。
- 启动命令：`uvicorn app.main:app --host 0.0.0.0 --port 8000`  
  - **`0.0.0.0`**：在容器**内部**监听所有网卡；若只写 `127.0.0.1`，宿主机浏览器会连不上。

### 5.3 `docker-compose.yml`（编排：数据库 + API）

- **`db`**：使用官方 **PostgreSQL** 镜像；数据放在卷 **`pgdata`** 里，容器删了数据一般还在。
- **`api`**：用当前目录的 `Dockerfile` **构建**并运行；把本机 **8000** 映射到容器 **8000**。
- **`DATABASE_URL`**：告诉程序数据库在哪。在 **api 容器里**主机名是 **`db`**（服务名），不是 `localhost`。
- **`depends_on` + `healthcheck`**：等数据库**就绪**再起 API，减少「一启动就连库失败」。

### 5.4 `.dockerignore`

避免把 `.venv`、`.git` 等大文件夹发给 Docker 构建进程，**加快构建、减小镜像无关内容**。

## 7. 起服务之后：Python 代码各自干什么

| 文件 | 一句话 |
|------|--------|
| `main.py` | 创建 `FastAPI` 应用；启动时 **建表**；把 **`/items`** 路由挂上去。 |
| `database.py` | 读环境变量 **`DATABASE_URL`**；创建 **引擎** 和 **会话工厂**；`get_db` 给每个 HTTP 请求一个数据库会话并在结束时关闭。 |
| `routers/items.py` | 把 URL（如 `POST /items/`）接到具体函数上；查不到时返回 **404**。 |
| `crud.py` | 真正执行 **增删改查**（`add` / `commit` / `query` 等），**不知道** HTTP。 |
| `models.py` | **表**长什么样（列名、类型）。 |
| `schemas.py` | **接口**上 JSON 长什么样（创建时要不要 `id`、返回时要不要 `created_at`）。 |

**为什么要分 `models` 和 `schemas`？**  
表结构会随时间变，对外接口往往要**隐藏内部字段**或**改名**；分开写更清晰，也和移动端 **DTO / Codable** 分层类似。

## 8. 数据库与增删改查（CRUD）

### 8.1 「数据库已经建好了吗？」

- **库与用户**：第一次启动 **`db` 容器**时，由环境变量自动创建（用户 `app`，库 `appdb`）。
- **表 `items`**：第一次启动 **API** 时，代码里 **`create_all`** 会根据 `models.py` 创建（若表已存在则不会清空数据）。

### 8.2 CRUD 和 HTTP 方法对应关系（背这个很有用）

| 想做的事 | HTTP 方法 | 本项目的路径（前缀都是 `/items`） |
|----------|-----------|-------------------------------------|
| **C**reate 创建 | POST | `POST /items/` |
| **R**ead 列表 | GET | `GET /items/` |
| **R**ead 单条 | GET | `GET /items/{id}` |
| **U**pdate 更新 | PATCH | `PATCH /items/{id}` |
| **D**elete 删除 | DELETE | `DELETE /items/{id}` |

### 8.3 创建时 JSON 长什么样？

```json
{
  "title": "必填字符串",
```

## 常见问题排查

### Docker 相关

| 现象 | 可能原因 | 试试 |
|------|----------|------|
| `Cannot connect to the Docker daemon` | Docker Desktop 没开 | 打开 Docker Desktop，等就绪后再执行命令 |
| `port is already allocated` | 8000 或 5432 被占用 | 关掉占用端口的程序，或改 `docker-compose.yml` 里 **左侧**端口（如 `"8001:8000"`），浏览器改用 8001 |
| 构建很慢 | 第一次要下镜像 | 正常；可换网络或使用镜像加速（视学校/地区而定） |

### API / 数据库相关

| 现象 | 可能原因 | 试试 |
|------|----------|------|
| 浏览器打不开 `/docs` | API 没起来或端口不对 | `docker compose ps`；看终端是否还在跑 `up`；URL 端口是否和 compose 一致 |
| `POST /items/` 报 500 | 数据库连不上 | 确认 `db` 在跑；看 `docker compose logs api` 里错误信息 |
| TablePlus 连不上 | db 没起或端口错 | `docker compose ps`；端口是否仍为 5432 |

### Python / venv 相关

| 现象 | 可能原因 | 试试 |
|------|----------|------|
| `command not found: uvicorn` | 没激活 venv 或没装依赖 | `source .venv/bin/activate` 后 `pip install -r requirements.txt` |
| Windows 无法 `activate` | 执行策略限制 | 以管理员身份开 PowerShell：`Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser` |

## 附录：云上 24 小时运行（概念与云厂商）

<a id="appendix-cloud-deploy"></a>

把服务从「自己电脑上的 Docker」搬到**云上长期运行**，本质是：**租一台（或多台）一直在网的机器/托管服务**，让你的 API 和数据库**有公网或稳定内网可访问**，且**不依赖你笔记本是否开机**。

---

### 1. 你要建立的心智模型

| 概念 | 通俗理解 |
|------|----------|
| **云厂商** | 提供机房、网络、虚拟机、数据库、存储、按量或包年计费的公司。 |
| **地域 / 可用区** | 服务器放在哪个城市、哪个数据中心；离用户近一般延迟更低；**跨地域**之间网络与计费都不同。 |
| **公网 IP** | 互联网上能找到你这台服务的「门牌」；**弹性公网 IP（EIP）** 可绑定到实例上，换机器时可迁移（视产品而定）。 |
| **安全组 / 防火墙** | **白名单**：例如只开放 **22**（SSH 登录）、**80/443**（网站/API）、数据库端口是否对公网开放要**极其谨慎**（生产常见做法：**数据库不对公网**，只给 API 内网连）。 |
| **带宽** | 出网流量速度与费用；小应用可先选低档，被攻击或突发流量会占满带宽。 |
| **镜像 / 快照 / 备份** | 系统盘、数据盘的备份策略；**数据库**务必有自动备份或定时导出。 |

**和你本机 Docker 的对应关系**：本机是「你的 Mac + Docker」；云上常见是「一台 Linux 云主机 + 安装 Docker + 同样 `docker compose up -d`」，或 **API 在云上、数据库用云厂商的托管数据库（RDS）**，不再自己维护 Postgres 容器。

---

### 2. 云服务分层（选型时经常听到）

| 类型 | 英文名 | 你得到什么 | 适合谁 |
|------|--------|------------|--------|
| **基础设施即服务** | IaaS | 虚拟机（vCPU、内存、磁盘），系统自己装 | 最灵活；会用 Linux、Docker 即可上手本项目 |
| **平台即服务** | PaaS | 上传代码或镜像，平台管运行时 | 省事，绑定厂商较强 |
| **容器编排** | K8s 等 | 多实例、滚动升级、服务发现 | 团队与流量大了再上 |
| **无服务器** | Serverless / FaaS | 按调用计费，冷启动 | 与当前「常驻 FastAPI 进程」模型不同，需改造 |

**对你当前项目最顺的入门路径**：**一台 Linux 云服务器（IaaS）+ Docker Compose**（与现在几乎一致）；进阶再把 **PostgreSQL 换成托管 RDS**，减轻备份与升级压力。

---

### 3. 部署形态（由简到繁）

1. **单机 Docker Compose（与开发最接近）**  
   在云主机上安装 Docker，上传项目，`docker compose up -d`。  
   **注意**：磁盘要做持久化（云盘挂载到卷路径），**重启实例**后数据仍在；安全组放行 **8000** 或前面加 **Nginx** 反代到 80/443。

2. **API 在容器 / 主机，数据库用托管 RDS**  
   云厂商提供 **PostgreSQL 兼容**的托管库，拿到内网地址与连接串，把 **`DATABASE_URL`** 改成云上的（用户/密码由控制台生成）。  
   **好处**：自动备份、补丁、监控；**compose 里可去掉 `db` 服务**，只起 `api`。

3. **前面加 Nginx + HTTPS**  
   用户与 App 使用 **`https://你的域名`**；Nginx 终止 TLS，反向代理到 **127.0.0.1:8000**。证书可用 **Let’s Encrypt**（免费，需域名）。

4. **域名与备案（中国大陆）**  
   若使用 **国内机房**且通过 **域名**对外提供 **网站类 HTTP/HTTPS 服务**，通常需要 **ICP 备案**（流程按云厂商指引，需时间）。  
   **仅 IP + 端口** 调试、或 **境外地域** 规则不同，以各云文档为准。

---

### 4. 与本项目衔接时要动什么

- **`DATABASE_URL`**：指向云上 Postgres（托管或自建容器），**勿**把生产密码写进 Git。  
- **`CORS`**：生产应把 `allow_origins=["*"]` 改为你的 **App / 前端域名**。  
- **`/docs`**：生产环境常**关闭或加鉴权**，避免接口被扫。  
- **日志与监控**：用 `docker compose logs`、或接入云监控、或 **Sentry** 等（进阶）。

### 8. 建议的落地顺序（自学路线）

1. 注册云账号，实名认证（国内云通常需要）。  
2. 买一台 **最小规格 Linux**（Ubuntu 22.04 LTS 等），配置 **安全组**（先只放行 SSH 与你的 API 端口）。  
3. SSH 登录，安装 Docker，**git clone** 或 scp 上传本项目，`docker compose up -d`。  
4. 浏览器访问 **`http://公网IP:8000/health`** 验证。  
5. 购买/解析 **域名**，上 **Nginx + HTTPS**；大陆业务再办 **备案**。  
6. 数据库迁到 **RDS** 或做好 **卷备份**，再考虑 **JWT、监控** 等。

更细的单家云「点击式」教程会随控制台改版而变，建议以 **该云官方文档：轻量/ECS + Docker + RDS** 为权威步骤。
